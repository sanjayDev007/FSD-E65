Code That Waits vs. Code That Doesn't üïê

Normal Code (Synchronous): Like standing in line at the school cafeteria - one step at a time, in order.
console.log("First in line"); console.log("Second in line"); console.log("Third in line"); // Output (in order): // First in line // Second in line // Third in line
Waiting Code (Asynchronous): Like ordering pizza, then playing a game while you wait for delivery.
console.log("Order pizza"); setTimeout(() => { console.log("Pizza delivered!"); }, 2000); // waits 2 seconds (2000 milliseconds) console.log("Play games while waiting"); // Output: // Order pizza // Play games while waiting // (after 2 seconds) Pizza delivered!

Real-life Example

Think of synchronous code like texting a friend and waiting for each reply before sending your next message.

Asynchronous code is like sending a text, then watching TikTok while waiting for the reply!

Making Code Wait with setTimeout ‚è≤Ô∏è

setTimeout is like telling JavaScript "Do this, but wait a bit first!"

setTimeout(whatToRun, howLongToWaitInMilliseconds); // 1000 milliseconds = 1 second

Basic Example

console.log("Starting game..."); setTimeout(() => { console.log("Game loaded! Let's play!"); }, 2000); // wait 2 seconds console.log("Loading assets...");
What you'll see:
Starting game...
Loading assets...
(2 seconds later)
Game loaded! Let's play!

Adding Extra Information

function showMessage(name, message) { console.log(`${name} says: ${message}`); } // Show message after 1 second setTimeout(showMessage, 1000, "Alex", "Hey, want to play Minecraft?");
Remember: setTimeout doesn't pause your code! Everything after it keeps running while it waits.

Cool setTimeout Tricks üéØ

Stopping a Timeout

Changed your mind? You can cancel a timeout before it happens:

// Set a delayed alert const timeoutId = setTimeout(() => { alert("Game Over!"); }, 5000); // Player scores a point, so we cancel the Game Over button.onclick = function() { clearTimeout(timeoutId); // Cancels the timeout alert("You scored! Game continues!"); };

Zero Delay?

Using setTimeout with 0 seconds doesn't run immediately - it runs after current code:

console.log("First"); setTimeout(() => { console.log("Third - even with zero delay!"); }, 0); console.log("Second"); // Output: // First // Second // Third - even with zero delay!
Fun fact: setTimeout times are not exact! They're more like "at least this long" rather than "exactly this long."

Repeating Code with setInterval üîÑ

setInterval is like saying "Do this over and over again, every X seconds."

setInterval(whatToRepeat, howOftenInMilliseconds);

Game Example

let score = 0; // Increase score every second const gameInterval = setInterval(() => { score += 10; console.log(`Score: ${score}`); if (score >= 50) { clearInterval(gameInterval); // Stop when score hits 50 console.log("Level complete! üéâ"); } }, 1000); // every 1 second
What you'll see:
Score: 10
Score: 20
Score: 30
Score: 40
Score: 50
Level complete! üéâ
Important: Unlike setTimeout which runs once, setInterval keeps running until you stop it with clearInterval()!

This is perfect for:

  • Game loops that update characters
  • Countdown timers
  • Animations
  • Updating scores

Callback Functions - Code That Runs Later üìû

A callback is a function you pass to another function, which will run it later.

Simple Example

function downloadGame(onComplete) { console.log("Downloading game..."); // Simulate download time setTimeout(() => { console.log("Download complete!"); onComplete(); // Run the callback function }, 3000); } downloadGame(function() { console.log("Let's play!"); }); // Output: // Downloading game... // (after 3 seconds) // Download complete! // Let's play!

Callback Problems

When you need many callbacks in a row, it gets messy:

// This is called "callback hell" - it's hard to read! downloadGame(function() { installGame(function() { loadLevel(function() { startGame(function() { console.log("Finally playing!"); }); }); }); });
Problem: This nested structure is like opening a bunch of boxes inside boxes - it gets confusing!

Promises - A Better Way to Wait ü§ù

A Promise is like a receipt for something that will happen in the future - like a pre-order for a video game.

Promises have three states:
1. Pending: Waiting for something to happen
2. Fulfilled: It worked! üòÑ
3. Rejected: Oops, something went wrong üòû

Making a Promise

const gamePromise = new Promise((resolve, reject) => { // Simulate loading a game setTimeout(() => { const gameLoaded = true; if (gameLoaded) { resolve("Game loaded successfully!"); // Success! } else { reject("Game failed to load!"); // Failure! } }, 2000); });

Using a Promise

gamePromise .then((message) => { console.log("Success:", message); return "Level 1 starting"; }) .then((levelMessage) => { console.log(levelMessage); }) .catch((error) => { console.log("Error:", error); });

Promise Chains - One After Another ‚õìÔ∏è

Promises can be linked together, so one happens after another - like steps in a video game level.

downloadGame() .then(result => { console.log("Game downloaded:", result); return installGame(); }) .then(result => { console.log("Game installed:", result); return loadLevel("Level 1"); }) .then(result => { console.log("Level loaded:", result); startGame(); }) .catch(error => { console.error("Something went wrong:", error); });

Social Media Post Example

uploadImage(myPhoto) .then(imageUrl => { // Image uploaded, now create the post return createPost(imageUrl, "Check out my cool photo!"); }) .then(post => { // Post created, now share it return sharePostWithFriends(post, ["Alex", "Taylor", "Jordan"]); }) .then(shareResult => { console.log(`Post shared with ${shareResult.count} friends!`); }) .catch(error => { console.error("Posting failed:", error); });
Cool trick: Each .then() passes its result to the next one in the chain!

Doing Things at the Same Time

// Load multiple things at once Promise.all([ loadCharacter(), loadBackground(), loadMusic() ]) .then(([character, background, music]) => { console.log("All game assets loaded! Ready to play!"); startGame(character, background, music); }) .catch(error => { console.error("Failed to load game:", error); });

Async/Await - Super Powers for Waiting ‚ú®

async/await is like magic that makes your waiting code look normal!

async: Tell JavaScript "this function will wait for things"
await: Tell JavaScript "pause here until this finishes"

Promises vs Async/Await

// Using Promises (more complex) function playGame() { return loadGame() .then(game => { console.log("Game loaded!"); return game.loadLevel(1); }) .then(level => { console.log("Level loaded!"); return level.start(); }); } // Using Async/Await (cleaner!) async function playGame() { const game = await loadGame(); console.log("Game loaded!"); const level = await game.loadLevel(1); console.log("Level loaded!"); return level.start(); }
It's still waiting! The code looks like normal code, but behind the scenes it's still waiting - just written in a cleaner way.

Async/Await Super Skills ü•ã

Handling Errors

async function loadPlayerProfile() { try { const player = await fetchPlayer("PlayerOne"); const stats = await fetchPlayerStats(player.id); const achievements = await fetchAchievements(player.id); return { player, stats, achievements }; } catch (error) { console.error("Oops! Something went wrong:", error); alert("Couldn't load profile. Are you connected to the internet?"); } }

Loading Multiple Things at Once

async function loadGameAssets() { try { // Load everything at the same time! const [characters, levels, music] = await Promise.all([ fetchCharacters(), fetchLevels(), fetchMusic() ]); console.log("All assets loaded! Ready to play!"); return { characters, levels, music }; } catch (error) { console.error("Failed to load game assets:", error); } }
Remember: You can only use await inside a function that has async in front of it!

Pro Tips for Better Code üß†

Always handle errors - What if loading fails? Always have a backup plan!
// Bad: No error handling fetch('https://game-api.example.com/player') .then(response => response.json()) .then(data => showPlayerData(data)); // Good: With error handling fetch('https://game-api.example.com/player') .then(response => { if (!response.ok) throw new Error('Network error'); return response.json(); }) .then(data => showPlayerData(data)) .catch(error => { console.error('Problem loading player:', error); showBackupPlayerData(); });

Top Tips Chart

Tip Why It's Important
Always handle errors Your game/app won't crash when something goes wrong
Use async/await for readability Your code will be easier to understand and debug
Clear your intervals when done Prevents memory leaks that slow down your game
Use Promise.all for loading multiple things Makes your game load faster by doing things at the same time
Keep timeouts reasonable Users get bored waiting more than a few seconds
